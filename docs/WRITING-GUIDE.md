# 文档写作规范

本规范约束 `docs/` 目录下所有文档的格式和风格，确保 AI 辅助撰写的文档与人工撰写的文档保持一致。

---

## 1. 语言

- **文档正文**：中文（简体）
- **代码注释**：英文
- **技术术语**：保留英文原文，不强行翻译（如 Agent、Memory、WebSocket、FTS5）
- **混排格式**：中英文之间加空格（如"使用 SQLite 存储"而非"使用SQLite存储"）

---

## 2. 文件组织

| 目录 | 用途 | 命名规则 |
|------|------|----------|
| `docs/` | 顶层规格文档 | `UPPER-CASE.md`（如 `DESIGN.md`, `FEATURES.md`） |
| `docs/plans/` | 功能实施方案 | `YYYY-MM-DD-kebab-topic.md` |
| `docs/devlogs/` | 开发日志 | `YYYY-MM-DD-kebab-topic.md` |
| `docs/todos/` | 想法/待办 | `YYYY-MM-DD-kebab-topic.md` |

## 3. Plan 文档结构

Plan 文档是最常写的文档类型。以下为标准结构：

### 3.1 Frontmatter（必须）

```yaml
---
title: "功能名称"
summary: "一句话描述做什么、解决什么问题"
read_when:
  - 场景 1（如"实施工具调用重试机制"）
  - 场景 2
  - 场景 3
---
```

### 3.2 标题 + 状态行

```markdown
# 功能名称

> 一段话概述背景和目标。
>
> 前置/关联文档引用：
> - `docs/plans/YYYY-MM-DD-xxx.md`（说明）
> - `packages/server/src/xxx.ts`（现有代码）
>
> **状态：✅ 全部完成（YYYY-MM-DD）** — 补充验证信息（如"Biome check 通过"）
```

状态行在完成后添加，未完成时省略。

### 3.3 需求总览表（必须）

紧接标题之后，一张表概括所有功能点：

```markdown
## 需求总览

| # | 功能 | 核心价值 | 工作量 | 优先级/状态 |
|---|------|---------|--------|-------------|
| 1 | 功能A | 解决什么问题 | N 天 | 🔴 关键 |
| 2 | 功能B | 解决什么问题 | N 天 | 🟠 重要 |
| 3 | 功能C | 解决什么问题 | N 天 | 🟡 增强 |
```

优先级标记：
- 🔴 关键 — 核心功能，必须实现
- 🟠 重要 — 显著提升体验，优先做
- 🟡 增强 — 锦上添花，按需做

完成后将优先级列改为 `✅`。

### 3.4 功能节（每个功能一节）

每个功能按以下子节组织：

```markdown
## N. 功能名称

### N.1 问题
描述当前痛点，简明扼要。

### N.2 方案
技术方案设计。包含：
- ASCII 架构图或流程图
- 关键接口的 TypeScript 代码片段
- 数据模型（SQL 建表语句）
- API 设计（方法 + 路径 + 说明表格）
- 前端设计（ASCII 线框图）

### N.3 实施步骤（表格）
| 步骤 | 内容 | 工作量 |
|------|------|--------|
| 1 | 具体任务 | Nh |
| 2 | 具体任务 | Nh |

### N.4 测试要点
- 验证场景 1
- 验证场景 2
```

### 3.5 尾部章节（必须）

```markdown
## 依赖关系与实施顺序
ASCII 箭头图，展示功能间的依赖和推荐并行分组。

## 影响范围汇总
| 文件 | 功能 | 变更类型 |
|------|------|---------|
| `path/to/file.ts` | #N | **新增** / 修改 |

## 总工作量估算
| 功能 | 后端 | 前端 | 合计 |
|------|------|------|------|

## 不做的事情
| 功能 | 理由 |
|------|------|
明确排除范围，防止 scope creep。

## 优于 OpenClaw 的设计（可选）
| 维度 | OpenClaw | YanClaw |
|------|----------|---------|
```

## 4. 格式约定

### 4.1 Markdown 标题层级

```
# H1 — 文档标题（每个文件仅一个）
## H2 — 功能编号或大章节
### H3 — 子节（问题/方案/步骤/测试）
#### H4 — 细节（代码块前的小标题）
```

### 4.2 代码块

- 始终标注语言：` ```typescript `、` ```sql `、` ```bash `、` ```json5 `
- 配置示例用 `json5`（项目使用 JSON5 格式，支持注释）
- 接口定义用 `typescript`，展示关键类型签名
- SQL 用 `sql`，展示建表语句
- ASCII 图用无语言标注的 ` ``` `

### 4.3 表格

- 对齐用途使用表格，不用无序列表
- 表头统一：`| 列名 |`，不加粗
- 常用表格模式：
  - **需求表**：# / 功能 / 核心价值 / 工作量 / 优先级
  - **步骤表**：步骤 / 内容 / 工作量
  - **文件表**：文件 / 功能 / 变更类型
  - **对比表**：维度 / OpenClaw / YanClaw
  - **API 表**：方法 / 路径 / 说明
  - **错误码表**：模块 / 错误码 / 说明

### 4.4 ASCII 图

架构图和流程图优先使用 ASCII box-drawing：

```
┌─────────┐     ┌─────────┐
│  组件 A  │────→│  组件 B  │
└────┬────┘     └─────────┘
     │
     ↓
┌─────────┐
│  组件 C  │
└─────────┘
```

前端线框图使用简化 ASCII：

```
┌─ 页面标题 ──────────────────────┐
│                                  │
│  [按钮]  🔍 搜索...  [类型 ▼]  │
│                                  │
│  ── 分组标题 ──────────────── │
│  │ 列表项 1              [操作] │
│  │ 列表项 2              [操作] │
│                                  │
└──────────────────────────────────┘
```

### 4.5 引用与交叉链接

- 引用其他文档：`` `docs/plans/YYYY-MM-DD-xxx.md` ``（反引号包裹完整路径）
- 引用代码文件：`` `packages/server/src/xxx.ts` ``
- 引用配置字段：`` `gateway.logging.level` ``
- 引用 API 端点：`` `GET /api/routing/bindings` ``

### 4.6 内联格式

- **粗体**：关键概念、节标题内的重点词
- `` `代码` ``：文件路径、变量名、API 路径、配置字段、命令
- *斜体*：不使用（中文斜体可读性差）
- 「」：不使用（统一用引号或粗体）

## 5. 内容原则

### 5.1 面向实施

- Plan 文档的目标是让开发者（或 AI）可以直接按文档实施
- 必须包含：关键接口代码、数据模型 SQL、API 路由定义、前端线框图
- 避免纯理论描述，每个"方案"节都应该有可执行的代码片段

### 5.2 工作量透明

- 每个功能节必须有**实施步骤表**，标注每步工时
- 文档末尾必须有**总工作量估算表**，按后端/前端拆分
- 工时单位：`h`（小时），汇总用天（1 天 ≈ 6h 有效工作）

### 5.3 明确边界

- **"不做的事情"** 节是必须的，防止 scope creep
- 每条都要写理由，不能只说"不做"
- 理由应简短直接（如"过度工程化"、"等反馈再做"、"规模不需要"）

### 5.4 可验证

- 每个功能节必须有**测试要点**
- 用具体场景描述，不用抽象指标（"用户发送消息后 Agent 回复中包含图谱实体"而非"准确率 > 85%"）

### 5.5 对比定位

- 涉及与 OpenClaw 竞争的功能时，补充"优于 OpenClaw 的设计"表
- 格式：`| 维度 | OpenClaw | YanClaw |`
- 客观描述差异，不夸大

## 6. Devlog 文档结构

开发日志记录已完成的工作，结构比 Plan 简单：

```markdown
# 标题

> 概述本轮完成的工作。
>
> 对照计划文档：
> - `docs/plans/YYYY-MM-DD-xxx.md`

## Phase N: 功能名称 ✅

- 完成项 1
- 完成项 2
- 涉及文件：`path/to/file.ts`（新增/修改）

## 检查结果

- Biome check: ✅ 通过
- 测试: ✅ N 个测试通过
```

---

## 7. FEATURES.md 结构

顶层功能文档使用 `F1-FN` 编号：

```markdown
## F1. 功能大类名称

### F1.1 子功能

描述 + 关键实现细节 + 配置示例。
```

## 8. Checklist

写完文档后自检：

- [ ] 有 frontmatter（plan 文档）
- [ ] 有需求总览表
- [ ] 每个功能有：问题 → 方案 → 步骤 → 测试
- [ ] 有影响范围汇总（列出所有新增/修改文件）
- [ ] 有总工作量估算
- [ ] 有"不做的事情"
- [ ] 有依赖关系图
- [ ] 代码块标注了语言
- [ ] 配置用 json5 格式
- [ ] 中英文之间有空格
- [ ] 无斜体（中文不用斜体）
