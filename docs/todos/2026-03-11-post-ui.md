## 需求 3：Onboarding 模型列表动态拉取

### 现状

| 问题 | 详情 |
|------|------|
| 模型列表硬编码 | `Onboarding.tsx:18-117` 中 `PROVIDERS` 常量包含固定模型 ID |
| 模型名过时 | 如 `claude-sonnet-4-20250514`、`gpt-4o` 等可能已有更新版本 |
| 无服务端模型列表 API | 服务端没有 `/api/models` 端点 |
| 用户体验繁琐 | 用户需要选供应商 → 填 API Key → 选模型，步骤较多 |

### 改进方案

#### 交互流程简化

```
当前：选供应商 → 填 API Key → 从硬编码列表选模型
改进：选供应商 → 填 API Key → 自动拉取可用模型列表 → 选模型
```

#### 需要的改动

**1. 服务端：新增模型列表 API**

路由：`GET /api/models/list`

请求参数：
```typescript
{
  providerType: "anthropic" | "openai" | "google" | "ollama" | "openai-compatible",
  apiKey: string,
  baseUrl?: string  // openai-compatible 需要
}
```

响应：
```typescript
{
  models: Array<{ id: string; name: string }>
}
```

实现逻辑：
- Anthropic：调用 `GET https://api.anthropic.com/v1/models`（需 API Key header）
- OpenAI：调用 `GET https://api.openai.com/v1/models`
- Google：调用 Gemini models list API
- Ollama：调用 `GET http://localhost:11434/api/tags`
- OpenAI-compatible：调用 `GET {baseUrl}/models`

> 注意：各供应商 API 返回格式不同，服务端需要统一适配。

**2. 前端：Onboarding Step 1 改造**

交互流程：

```
选择供应商（卡片网格）
    ↓
输入 API Key（+ Base URL，如需要）
    ↓ 点击"验证并获取模型"
调用 /api/models/list，显示加载状态
    ↓ 成功
显示可用模型列表（下拉选择）
    ↓ 失败
显示错误提示，允许重试或手动输入模型 ID
```

关键改进点：
- 移除硬编码 `PROVIDERS[].models` 数组
- API Key 输入后自动验证 + 拉取模型
- 拉取失败时 fallback 到手动输入（兼容离线/自建场景）
- 加载状态用 Skeleton 组件（已安装但未使用）

**3. Settings 页模型选择同步改造**

Settings 页的 System Models 部分也引用模型 ID，应复用同一模型列表获取逻辑。

### 涉及文件

| 文件 | 操作 |
|------|------|
| `packages/server/src/routes/models.ts` | **新建**：模型列表 API 路由 |
| `packages/server/src/app.ts` | **修改**：注册新路由 |
| `packages/web/src/pages/Onboarding.tsx` | **修改**：动态拉取模型 |
| `packages/web/src/pages/Settings.tsx` | **修改**：复用模型拉取逻辑 |
