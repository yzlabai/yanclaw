# 工作目录（WorkDir）统一管理需求分析

## 背景

YanClaw 的数据文件分散在 `~/.yanclaw` 下，Server 端已通过 `resolveDataDir()` + `YANCLAW_DATA_DIR` 环境变量实现了可配置，但存在以下问题：

1. Tauri 端硬编码 `~/.yanclaw`，未读取环境变量
2. 用户无法在 UI 上看到当前数据目录
3. Agent 人格配置只能通过 config 的 `systemPrompt` 字符串，不支持 SOUL.md 文件
4. 启动日志未提示当前工作目录路径

## 目标

- 支持用户指定一个统一的工作目录，所有数据、配置、SOUL.md 等文件集中存放
- 不指定时使用默认路径 `~/.yanclaw`
- 用户可在 UI 上看到当前工作目录路径

## 现状分析

### 数据目录下的文件布局

```
{dataDir}/                    # 默认 ~/.yanclaw
├── config.json5              # 主配置文件
├── data.db                   # SQLite 数据库
├── auth.token                # 认证令牌（自动生成）
├── vault.json                # 加密凭据存储
├── server.log                # 服务器日志（Tauri 模式）
├── media/                    # 上传/缓存的媒体文件
├── plugins/                  # 插件目录
└── workspace/                # Agent 工作区
    └── {agentId}/            # 每个 agent 的工作目录
```

### Server 端（已完成 ✅）

`resolveDataDir()` 在 `packages/server/src/config/store.ts:9`：

```typescript
export function resolveDataDir(): string {
    return process.env.YANCLAW_DATA_DIR ?? join(homedir(), ".yanclaw");
}
```

所有使用 `resolveDataDir()` 的模块：

| 模块 | 文件 | 用途 |
|------|------|------|
| ConfigStore | `config/store.ts` | 配置文件路径、auth.token |
| Database | `db/sqlite.ts` | data.db 路径 |
| MediaStore | `media/store.ts` | media/ 目录 |
| PluginLoader | `plugins/loader.ts` | plugins/ 目录 |
| AgentRuntime | `agents/runtime.ts` | workspace/{agentId}/ |
| TokenRotation | `security/token-rotation.ts` | auth.token |
| VaultMigrate | `security/vault-migrate.ts` | vault.json |
| CredentialVault | `security/vault.ts` | vault.json（注意：此处有独立的 `YANCLAW_DATA_DIR` 读取） |

**结论：** Server 端路径管理已统一，无需大改。

### Tauri 端（需要修改 ❌）

`src-tauri/src/lib.rs:22-25` 硬编码了路径：

```rust
fn data_dir() -> PathBuf {
    let home = dirs_home();
    home.join(".yanclaw")
}
```

未读取 `YANCLAW_DATA_DIR` 环境变量，与 Server 端不一致。

### API 层（缺失 ❌）

`/api/system/status` 和 `/api/system/setup` 均未返回 `dataDir` 信息，前端无法获知当前工作目录。

### 前端 UI（缺失 ❌）

没有任何地方展示当前工作目录路径。

### SOUL.md 机制（缺失 ❌）

Agent 人格只能通过 `config.json5` 的 `systemPrompt` 字段配置（纯字符串），不支持从文件加载。

---

## 改动方案

### 1. Tauri 端同步环境变量

**文件：** `src-tauri/src/lib.rs`
**改动量：** 小

```rust
fn data_dir() -> PathBuf {
    if let Ok(dir) = std::env::var("YANCLAW_DATA_DIR") {
        return PathBuf::from(dir);
    }
    dirs_home().join(".yanclaw")
}
```

同时新增 IPC 命令 `get_data_dir()` 供前端查询。

### 2. API 暴露 dataDir

**文件：** `packages/server/src/routes/system.ts`
**改动量：** 小

在 `/api/system/status` 响应中添加 `dataDir` 字段：

```typescript
dataDir: resolveDataDir(),
```

### 3. Server 启动日志

**文件：** `packages/server/src/gateway.ts` 或 `index.ts`
**改动量：** 小

启动时打印：

```
[gateway] Data directory: /home/user/.yanclaw
```

用户一眼就能确认数据存储位置。

### 4. 前端展示工作目录

**文件：** 状态页或设置页组件
**改动量：** 小

在系统状态信息中展示 `dataDir` 路径，并标注是否为默认路径。示例：

```
工作目录: ~/.yanclaw (默认)
工作目录: /data/yanclaw (自定义)
```

### 5. SOUL.md 文件加载

**文件：** `packages/server/src/agents/runtime.ts`
**改动量：** 中

#### 设计

- 加载路径（按优先级）：
  1. `{dataDir}/workspace/{agentId}/SOUL.md` — agent 专属人格
  2. `{dataDir}/SOUL.md` — 全局人格
- 与 `config.systemPrompt` 的关系：**合并而非替代**
  - 最终 systemPrompt = `SOUL.md 内容` + `\n\n` + `config.systemPrompt`
  - 如果 SOUL.md 不存在，行为不变
- 文件变更时热重载（利用现有 config watcher 机制或新增 watcher）

#### 文件布局变更

```
{dataDir}/
├── SOUL.md                   # 全局人格文件（可选）
├── workspace/
│   └── {agentId}/
│       └── SOUL.md           # Agent 专属人格（可选，优先）
```

#### 为什么是合并而非替代

- `SOUL.md` 适合写长篇人格描述、知识库、行为准则
- `config.systemPrompt` 适合写简短的角色定义
- 两者互补，用户可以灵活选择只用一个或两个都用

## 改动影响评估

| 改动项 | 影响范围 | 风险 | 工作量 |
|--------|---------|------|--------|
| Tauri data_dir() | 仅 Rust 端 | 低 — 逻辑简单 | 小 |
| API 暴露 dataDir | 仅 system route | 低 — 只加字段 | 小 |
| 启动日志 | 仅启动流程 | 无 | 极小 |
| 前端展示 | UI 组件 | 低 | 小 |
| SOUL.md 加载 | Agent runtime | 中 — 需处理文件不存在、编码、热重载 | 中 |

**总工作量：** 中等偏小，无架构变更，所有改动都是在现有框架上补齐。

## 不需要改的

- `resolveDataDir()` 函数本身 — 已经支持环境变量
- 数据库、媒体、插件等路径 — 已统一走 `resolveDataDir()`
- 配置文件格式 — 不需要在 config.json5 里加 dataDir 字段（由环境变量控制即可）

## 后续可考虑

- **UI 中直接切换工作目录**：需要重启 server，可在 Tauri 端实现"修改 env → 重启 gateway"流程
- **多工作目录 profile**：类似 VS Code 的 workspace 概念，暂不需要
