# YanClaw 开发迭代流程规范

> 本文档定义了 YanClaw 项目的标准开发迭代流程。无论是人工开发还是 AI 辅助开发（Claude Code `/dev-iterate`），都应遵循此规范。

---

## 核心原则

| 原则 | 说明 |
|------|------|
| **最小变更** | 只改需要改的。不顺手重构、不提前抽象、不加"以防万一"的错误处理 |
| **模式一致** | 新代码必须遵循现有模式。引入新范式需要有充分理由 |
| **安全优先** | 每个系统边界必须有输入校验。遵循 OWASP Top 10 |
| **类型贯穿** | 维护 Hono → AppType → hc 的端到端类型链 |
| **验证门控** | format → check → build → test 四步全过才算完成 |

---

## 迭代流程

### Phase 0 — 上下文加载

开始任何工作之前：

1. **了解项目现状**：阅读 `CLAUDE.md`（架构全景）和 `docs/CHANGELOG.md`（最近进展）
2. **了解相关代码**：找到并阅读将要修改的模块及其上下游依赖
3. **了解历史决策**：如有疑问，查看 `git log` 了解相关模块的演进原因

> **门控条件**：能说清楚"改哪些文件、为什么这样改、会影响什么"才进入下一步。

---

### Phase 1 — 设计

目标：用最简方案解决问题。

#### 1.1 范围界定

- 列出需要修改的文件清单
- 明确什么**不做**（避免范围蔓延）
- 如果涉及 >3 个文件或架构变更，写简要设计说明

#### 1.2 方案选择

优先级：
1. 改现有代码 > 新建文件
2. 用现有模式 > 引入新模式
3. 简单直接 > 优雅抽象

#### 1.3 风险评估

| 检查项 | 说明 |
|--------|------|
| 类型链影响 | 改了路由？AppType 是否自动更新？前端需要改吗？ |
| 配置兼容性 | 新增 config 字段有默认值吗？老配置会报错吗？ |
| 数据库迁移 | 改了 schema？需要迁移脚本吗？ |
| 安全影响 | 新端点需要认证吗？输入有校验吗？ |
| 通道影响 | 改了消息处理？所有通道适配器都兼容吗？ |

#### 1.4 任务拆分

将工作拆为可独立验证的小步骤。每步完成后应该可以通过验证门控。

> **门控条件**：非平凡变更（架构调整、新模块、跨包改动）应先与团队对齐方案再动手。

---

### Phase 2 — 实现

#### 2.1 实现顺序

```
Schema / 类型  →  后端逻辑  →  API 路由  →  前端 UI  →  接线（app.ts / gateway.ts）
```

遵循此顺序可以确保：
- 类型先就位，后续代码有编译期保障
- 后端逻辑独立于 HTTP 层，便于测试
- 前端最后做，自动获得完整的类型推断

#### 2.2 代码规范

**必须**：
- Biome 格式：Tab 缩进、100 字符行宽、双引号、始终分号
- 新 API 输入使用 `zValidator('json', zodSchema)`
- 路径别名 `@yanclaw/server/*`、`@yanclaw/web/*`、`@yanclaw/shared/*`
- 错误消息可操作（告诉用户该怎么做，而不只是报错）
- 日志使用 Pino（`log.agent()`、`log.channel()` 等），不用 `console.*`

**禁止**：
- `// @ts-ignore` 或 `// biome-ignore`（修根因，不压警告）
- `console.log` / `debugger`（调试完必须清理）
- 硬编码魔法值（应放入 config schema 或常量）
- 提前抽象（三行重复代码比一个过早的 helper 好）
- 兼容性垫片（`_unused` 变量、`// removed` 注释）

#### 2.3 进度跟踪

- 按任务拆分逐个完成，完成一个标记一个
- 发现计划有误时，先更新计划再继续实现

---

### Phase 3 — 验证

**四步门控**，按顺序执行，全部通过才算完成：

```bash
# 1. 格式化（自动修复）
bun run format

# 2. 代码检查（Biome lint）
bun run check

# 3. 类型检查 + 构建
bun run build

# 4. 测试
bun run test
```

#### 验证规则

| 规则 | 说明 |
|------|------|
| 修根因 | lint/类型错误必须修复，不能压制 |
| 全量通过 | 四步全部 pass。不能因为"改动很小"就跳过 |
| 不破现有 | 已有测试不能因新代码而失败（除非测试预期需要随业务更新） |
| 循环修复 | 如果 step 3 失败，修完后从 step 1 重新跑 |

#### 测试分层

| 层级 | 命令 | 何时运行 |
|------|------|----------|
| 单元/集成 | `bun run test` | 每次提交前（必须） |
| 手动冒烟 | `bun run dev:server` + 手动操作 | 涉及 UI 或通道交互时 |
| 端到端 | （未来）`bun run test:e2e` | CI / 发版前 |

---

### Phase 4 — 自审

在声明完成之前，审查自己的改动：

```bash
git diff  # 查看所有变更
```

#### 审查清单

- [ ] **调试残留**：无 `console.log`、`debugger`、`TODO: remove` 等
- [ ] **安全**：新端点有认证？输入有校验？无 XSS/注入/SSRF 风险？
- [ ] **类型链**：路由变更后 `AppType` 正确导出？前端 `hc<AppType>()` 能推断？
- [ ] **配置**：新 config 字段有 Zod 默认值？schema.ts 和 docs 同步？
- [ ] **向后兼容**：现有用户配置不会因更新而崩溃？
- [ ] **导入路径**：使用 `@yanclaw/*` 别名，不用相对路径跨包引用
- [ ] **错误消息**：用户能看懂并知道该怎么做？

---

### Phase 5 — 文档更新

**仅在变更对用户可见时更新文档**：

| 文档 | 何时更新 |
|------|----------|
| `docs/CHANGELOG.md` | 新功能、重要修复、破坏性变更 |
| `CLAUDE.md` | 新增架构模式、关键模块、约定 |
| `docs/DEVELOPMENT.md` | 新增开发命令或工具链变更 |
| `docs/*.md` 专题文档 | 新增配置项、API、安全特性 |

**不要**：
- 为未改的代码补文档
- 创建新的 README 文件（除非明确要求）
- 给已有函数补 JSDoc/注释

---

### Phase 6 — 提交

#### 提交信息格式

```
<type>: <description>

[optional body]
```

**type 类型**：

| type | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构（不改行为） |
| `perf` | 性能优化 |
| `docs` | 仅文档 |
| `chore` | 构建/依赖/配置 |
| `test` | 仅测试 |

**规则**：
- 描述写"为什么"而不是"改了什么"（diff 已经说了改了什么）
- 一个功能一个提交（不混装不相关的改动）
- 中文或英文均可，保持一致
- 不跳过 pre-commit hooks

## 版本发布流程

```
1. 更新 VERSION 常量 + package.json 版本号
2. 更新 CHANGELOG.md
3. 完整验证门控（format → check → build → test）
4. Git tag（格式 v0.x.x）+ push
5. 构建桌面安装包（bun run dev:tauri build）
6. 创建 GitHub Release
```

详见 [DEVELOPMENT.md § 发版流程](DEVELOPMENT.md#发版流程)。

## 错误恢复策略

| 阶段 | 卡住时怎么做 |
|------|-------------|
| 设计 | 需求不明就问，不要猜 |
| 实现 | 同一方案尝试 2 次不通就回退重新设计 |
| 验证 | 无关的测试失败：记录后跳过；相关的：必须修 |
| 自审 | 发现重大问题：回到实现阶段修复后重新验证 |

## 反模式清单

以下做法在本项目中**明确禁止**：

- 不读代码就改代码
- 加入未被要求的功能
- 顺手重构未涉及的代码
- 给显而易见的代码加注释
- 为一次性操作创建工具函数
- 加兼容性垫片而不是直接改代码
- 因为"改动小"就跳过验证
- 用 `// @ts-ignore` 压制类型错误
- 在一个提交中混装多个不相关的改动
- 未经确认就执行不可逆操作（force push、drop table、rm -rf）
